# Revenue Ops

Status: partial until RevenueCat, store products, entitlements, and sandbox proof are recorded.

## Definition Of Good

### PRESENT — artifact exists (required before any revenue claim)

- `REVENUE_OPS.md` (this file) documents the chosen paywall model, product mix, entitlement name, and offering ID.
- Every App Store and/or Google Play product is listed with its store product ID, RevenueCat product ID, RevenueCat product type, entitlement membership, and package ID.
- A `currentOffering` is identified by name (default: `default`) and every package it contains is listed.
- `PROVIDER_PROOF.md` contains a RevenueCat row with an `evidence path` referencing a proof artifact (e.g. `revenue/revenuecat-proof.md`).
- Paywall model decision is recorded: hard paywall, soft paywall/freemium, reverse trial, or web funnel, with the value moment shown before the paywall.
- Trial/intro-offer duration, plan mix (weekly/monthly/annual/lifetime), and price are recorded as explicit decisions.

### PROVEN — hard errors when revenue lane is "done"

All items below are hard errors (exit 1) when `lanes.revenue.status` is `"done"`:

1. **Founder-gated live probe artifact (`revenue/revenuecat-proof.json`) exists on disk.**
   The file must be generated by running:
   ```
   doppler run -- npm run probe:revenuecat
   ```
   It must carry a `"probe":"revenuecat@1"` fingerprint, a fresh ISO timestamp
   (warn if older than 30 days), a non-empty `offering_id`, and at least one
   `entitlement_id`. A hand-typed copy of the example file fails the Tier-1
   content floor (byte-identity check). Prose/keyword scans of a markdown file
   are no longer accepted as the primary offering check when the proof JSON is
   present; the JSON artifact is the authoritative proof.

2. **Entitlement unlock confirmed.** The proof artifact or `PROVIDER_PROOF.md`
   entry must record that a sandbox purchase granted the named entitlement
   inside the app — not just that the purchase appeared in the RevenueCat
   dashboard. Record this in `revenue/revenuecat-proof.md` (the human-readable
   companion stub).

3. **No MISSING_METADATA.** No App Store subscription product listed in
   `REVENUE_OPS.md` may be in `MISSING_METADATA` state. Subscription-group
   localizations must be confirmed created before the lane is marked done.

4. **Product-type reconciliation.** Every product listed in `REVENUE_OPS.md`
   must declare its RevenueCat product type, and that type must match the App
   Store/Google Play product type: `non_consumable` for lifetime/one-time
   unlocks; `auto_renewable_subscription` for recurring subscriptions. A
   lifetime product mapped as `non_renewing_subscription` is a hard error.

5. **Restore tested.** The proof artifact or `PRODUCTION_READINESS.md` must
   record that the restore-purchases path was exercised without granting a
   duplicate entitlement.

### OPTIMIZED — warnings (taste stays human; logged override accepted)

- Paywall model, trial duration, and price are documented with a benchmark citation (e.g. RevenueCat State of Subscription Apps 2026) explaining why this combination was chosen rather than a default. Absence is a warning, not an error.
- Annual plan is offered as the highlighted option alongside monthly/entry; leading with monthly only triggers a warning.
- Hard paywall or value-moment decision is recorded with a rationale; choosing freemium without documenting the word-of-mouth / network-effect justification triggers a warning.
- Involuntary-churn recovery (grace period, billing-issue webhook → dunning) is addressed in the engineering plan or noted as deferred with a reason.

## Founder-Gated Probe Step

The revenue lane requires a live RevenueCat API probe to confirm offerings and
entitlements resolve correctly. This probe is founder-gated because it requires
live credentials injected via Doppler.

### Running The Probe

```
doppler run -- npm run probe:revenuecat [-- --root /path/to/business-repo]
```

Required Doppler secrets:
- `REVENUECAT_SECRET_API_KEY` — RevenueCat secret key (starts with `sk_`)
- `REVENUECAT_PROJECT_ID` — RevenueCat project ID (visible in dashboard URL)

### What The Probe Does

1. Calls `GET /v2/projects/{project_id}/offerings` and records the default
   offering ID and package IDs.
2. Calls `GET /v2/projects/{project_id}/entitlements` and records entitlement IDs.
3. Writes a non-secret proof artifact at `revenue/revenuecat-proof.json` with a
   `"probe":"revenuecat@1"` fingerprint, HTTP statuses, offering ID, package IDs,
   entitlement IDs, and an ISO timestamp.

No API key or secret is written to the artifact.

### After The Probe

1. Review `revenue/revenuecat-proof.json` to confirm `offering_resolved: true`
   and `entitlements_present: true`.
2. Open `revenue/revenuecat-proof.md` and fill in the sandbox purchase, entitlement
   grant, restore, and Release-build evidence sections.
3. The validator (`npm run check:revenue`) will verify the JSON artifact's shape,
   fingerprint, timestamp freshness, and non-empty offering/entitlement values.
   The validator also applies a Tier-1 content floor: if the JSON is byte-identical
   to the example file or below a minimum size threshold, it is rejected.
