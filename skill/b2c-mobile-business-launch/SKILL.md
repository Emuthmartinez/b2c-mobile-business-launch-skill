# B2C Mobile Business Launch

This skill turns an app idea into a launchable business package: evidence, positioning, 11-star experience, design, ASO, implementation handoff, pre-launch funnel, analytics, and verification. It should also leave a machine-readable `PROJECT_STATE.yaml`, founder-visible `launch-cockpit.html`, a versioned Design Room over `state/business.json`, and LaunchBench/validator proof for known failure modes.

The model pattern is evidence-first: AppKittie/category economics first, social-language research second, canonical docs third, then a builder-ready bundle and a live waitlist or purchase funnel. Keep the workflow generic; do not copy names, prices, tokens, domains, or credentials from prior launches unless the current project explicitly owns them.

## Runtime Freshness Gate

Before substantial launch, design, store, revenue, or build work, check whether the installed skill runtime is behind the latest available source copy. Run `npm run check:skill-version` from the installed skill when available, or compare `skill-version.json` manually if the helper is unavailable.

If the installed runtime is stale, pause and ask the founder — AskUserQuestion when available, plainly otherwise — whether to update now or continue on the installed version; never silently continue stale unless the founder declines or the latest source copy is unavailable. [`references/skill-versioning.md`](references/skill-versioning.md) carries the ask flow, commands, and local sync rules.

## Design Room Contract

For all design, visual-system, cross-surface, App Store creative, landing, onboarding, paywall, and marketing-surface work, use **STATE -> MUTATE -> VERSION -> RENDER**:

1. **STATE**: read `state/business.json` and `state/theme.tokens.json`. If they are missing, seed them from the skill's `state/` or `templates/state/`.
2. **MUTATE**: make one coherent JSON state mutation. Do not invent a one-off design proposal document or ad-hoc HTML proof.
3. **VERSION**: validate, render, and version the state with git. Baselines are `git tag baseline/<name>`; diffs and restores operate on `state/`.
4. **RENDER**: show `design-room.html` or the React/Vite build in `dist/design-room/`. The renderer reads state and tokens; it is the committed visual medium.

The user reviews the rendered Design Room. The agent edits the state. Load [`references/design-room.md`](references/design-room.md), [`references/surfaces-b2c.md`](references/surfaces-b2c.md), [`references/quality-lens.md`](references/quality-lens.md), and [`references/control-plane.md`](references/control-plane.md) when design or control-plane work is in scope.

When theme tokens change and the design is accepted, run the token promotion path so `state/theme.tokens.json` flows into app-facing `design-system/` artifacts before implementation relies on the visual system.

## Autopilot Run Contract

When this skill activates for broad launch/business work, keep running the launch workflow without asking the user to re-invoke this skill. Load the next needed reference yourself, update `PROJECT_STATE.yaml`, render `launch-cockpit.html`, and run the relevant validators before readiness claims. Do not stop with instructions.

Assume a beginner founder. Load [`references/founder-zero-operator.md`](references/founder-zero-operator.md), seed access state, present one plain-language action, do the rest, and continue when access clears.

**Founder Question Contract:** Follow `founder-zero-operator.md`: current phase/outcome, defined terms, two or three choices, recommendation, consequences, next action, Skip or defer, and fallback. Use AskUserQuestion or plain-text fallback. Protected gates only defer; silence does not grant consent; direct instructions supersede stale gates.

Only pause for founder-only gates: credentials, secrets, account access; spend; legal/pricing; domains; public actions; destruction; or final submission/release decisions. Before browser/provider/social/device work, load [`references/frontier-agent-operations.md`](references/frontier-agent-operations.md), scope approval, and reconcile its ledger, state, and cockpit.

Provider-backed readiness must include live proof or explicit founder-only gates in `PROVIDER_PROOF.md`; setup prose alone cannot mark analytics, revenue, email, store, security, or engineering lanes done. Load [`references/provider-proof.md`](references/provider-proof.md) when any provider-backed lane approaches readiness.

## Claude vs Codex Routing And Dynamic Workflows

This skill recommends a default bias — Claude for the pre-build stages through the spec, Codex for the core app build — but it is a recommendation, not a hard requirement. Either runtime can do any stage; the founder decides, and the skill never blocks, gates, or refuses work in whichever runtime is in front of it. When the founder leans this way, Claude takes each research, experience, design, growth, analytics, and spec stage to a locked, evidence-backed artifact, then hands a complete spec bundle to Codex (or Compound Engineering on Codex) for the build. Record which runtime is handling which lane in `ORCHESTRATION.md` and `PROJECT_STATE.yaml` so later sessions do not re-litigate it.

Catch the Codex case: when the runtime is Codex (or any non-Claude-Code runtime, detectable because `ultracode`/`/workflows`/`/deep-research` are unavailable) and the active work is a pre-build stage, surface the recommendation **once** — plainly, not as a founder-only gate — that those stages tend to go better on Claude Code with Dynamic Workflows, give the option to switch or stay, record that it was surfaced in `PROJECT_STATE.yaml`, then continue the work in the current runtime regardless. Do not nag, do not pause, and never downgrade quality: if staying in Codex, run the same fan-out/adversarial-verification/quarantine shapes as subagents or inline.

For non-trivial **pre-build** stages that are long-running, massively parallel, highly structured, or adversarial — deep research/social mining, 11-star/emotional design, analytics, paid UA, viral growth, launch narrative, localization market research, design/taste exploration, naming, and spec/handoff readiness — prefer a **Dynamic Workflow** when running Claude Code: trigger with `ultracode` (or `/effort ultracode`, or `/deep-research`), set a token budget, pair loop patterns with `/goal`, quarantine any untrusted public content (reviews, XPOZ/social, scraped pages), and keep producer and verifier agents separate. Do not spend a Claude workflow on the core engineering build when Codex owns that lane; workflows on the Claude side stop at the spec. Load [`references/dynamic-workflows.md`](references/dynamic-workflows.md) before proposing or running a workflow, and fall back to [`references/parallel-agent-orchestration.md`](references/parallel-agent-orchestration.md) subagents (recording the reason) when workflows are unavailable, disabled, or below Claude Code v2.1.154. Do not reach for a workflow when a normal session would finish the stage in minutes.

26. **Evaluate known failure modes.** Use [`references/launchbench-evals.md`](references/launchbench-evals.md), [`references/failure-cards.md`](references/failure-cards.md), and bundled `scripts/` before claiming launch readiness or after major skill/app changes. Known misses should become failure cards or LaunchBench scenarios, not oral lore.
27. **Stop only at real gates.** Ask the founder for access, payments, domain purchases, credential changes, pricing approval, legal approval, destructive repo actions, connecting social accounts, public posting/scheduling, or final ship/merge decisions. Resolve bounded strategy/design choices yourself, preferably with `llm-council` when available.

## Operating Posture

- Evidence beats taste. Category, pricing, keywords, social language, and moat claims need App Store, competitor, review, XPOZ/social, or live funnel evidence.
- Design the extreme before cutting scope. The 11-star ladder is not a polish exercise; it is how the product chooses the one magical V1 moment that design, engineering, store, ads, email, and support must carry.
- Charge every user-facing moment before specs harden. The Experience Cards (Commitment, Variable Reward, Perceived Effort Delay, Intent Mirroring + the deck) are mechanics that fill the 11-star 6/7-star levels, not decoration — each must carry a PostHog event, a bright-line guardrail, and a reduced-motion fallback before build handoff, or the emotional_design lane is incomplete. Engineered emotion that serves the user's real goal builds durable retention; the same mechanics aimed at extraction are dark patterns and a compliance veto.
- Lock phase outputs before depending on them. Do not build design from an unlocked spec, ASO from an unlocked name, or a landing page from drifting pricing/voice.
- Recommend the runtime split, do not enforce it. The default bias is Claude for the pre-build stages through the spec (using Dynamic Workflows for the long-running, parallel, or adversarial ones) and Codex for the core app build — but it is a recommendation, not a gate. When a Codex (or other non-Claude-Code) runtime is doing pre-build work, surface the Claude-for-pre-build tip once, record it in `PROJECT_STATE.yaml`, and continue regardless. Record which runtime handled which lane in `ORCHESTRATION.md`; do not burn a Claude workflow on the build Codex is better at, and never skip the adversarial-verification/quarantine shape just because the workflow runtime is unavailable.
- Scope before producing. Confirm the launch tier at orient and let lite launches defer breadth lanes with dated reasons; thirty artifacts on a quick-test utility is overproduction, and silent lane-skipping is worse — the tier makes scope visible to validators.
- Keep `PROJECT_STATE.yaml` current. It is the compact state contract validators, subagents, and future sessions read instead of re-reading every doc; the Deliverable Standard below enumerates what it must carry.
- Treat session continuity as a validator-backed contract. New sessions, resumes, status checks, and handoffs must reconstruct state from durable files, route broad work through `APP_AGENTS.md` role prompts or record the unavailable-subagent rationale, and leave the next action in state before pausing.
- Do not silently downgrade paid/account-gated tooling. Missing runtime access means ask, wait for access/export, or use a founder-approved fallback.
- Keep third-party guidance fresh. New external sources must be registered in `source-registry.yaml`; fast-moving commands require current docs or local CLI help before agents repeat them.
- Preserve a clear source of truth. Every launch should leave an `AGENTS.md` or equivalent agent entrypoint, plus product, brand, design, analytics, launch, and prompt/handoff docs where appropriate.
- Treat public claims as liabilities. Avoid unsupported endorsement, revenue, neuroscience, health, urgency, scarcity, and pricing claims.
- Treat security as a release lane. Do not call an app launch-ready from working screens alone; threat model, secret routing, platform hardening, entitlement/webhook abuse controls, monitoring, and incident response need evidence or explicit blockers.
- Verify what shipped. For landing and funnel work, run local build, deploy checks, live HTTP checks, form submission smoke tests, analytics event verification, Firecrawl/web-crawl checks where useful, GEO/SEO and AI-crawler checks, and mobile/desktop visual QA.
- Cascade every change. A change to a launched app's feature, copy, brand vocabulary, pricing, products, or data behavior rarely lives on one surface. After any such change, run [`references/change-cascade.md`](references/change-cascade.md) to propagate it across in-app, App Store listing/keywords/screenshots/App Preview/products/localizations, RevenueCat, landing page + meta/JSON-LD, SEO/GEO, email, analytics, and legal — re-rendering derived assets and reconciling the lexicon before calling the change done.
- Treat App Store Connect access as setup, not a wall. If the `asc` CLI is installed, never report "cannot access ASC"; run the auth-recovery ladder in [`references/app-store-connect-cli.md`](references/app-store-connect-cli.md) (keychain profiles, account-level keys, `asc auth init/login`) and report any missing app record/cert/RevenueCat app as a founder-gated setup step with the exact next command.
- Treat design versioning as git state, not prose labels. For Design Room work, baselines, diffs, restores, and wipes operate on `state/` and the rendered Design Room.
- Treat Control Plane, the Business Control workspace read model, live-provider proof, behavior evals, artifact starters, and token promotion as mechanically enforced contracts; run their validators before claiming the skill or a generated launch package is complete.

## Default Phase Flow

1. **Phase 0: founder-zero orient and scaffold** - source inventory, `BUSINESS_ACCESS.md` plus ledger, business identity/access/Doppler next action, `PROJECT_STATE.yaml`, autonomy/tier, and first cockpit. The agent drives; the founder gets one plain-language gate.
2. **Phase 0b: paid-tool access and fallback routing** - identify paid/account-gated lanes, ask before free fallbacks, record `TOOL_DECISIONS.md`, and separate blocked access from approved fallback work.
3. **Phase 0c: secrets baseline** - create `SECRETS.md`, select Doppler or approved alternative, map local/staging/production secret locations, add `doppler.yaml` and `.env.example` names only when useful, and route new secrets as they appear.
4. **Phase 1: research-backed spec** - category economics, competitor deep dives, review/social language, keyword scan, name collision, localization market opportunity matrix (research demand per storefront before localizing; `LOCALIZATION_MARKET_RESEARCH.md`), revised spec with evidence ledger.
5. **Phase 1b: analytics and attribution blueprint** - current PostHog doc map, identity plan, UTM/referrer/deep-link/self-reported attribution, event catalog, dashboards, experiment plan, privacy guardrails, QA plan, `analytics-plan.html`.
6. **Phase 1c: 11-star experience and product brainstorm checkpoint** - create `11_STAR_EXPERIENCE.md` and `11-star-experience.html`, define the 1/2/5/6/7/10/11-star ladder, draw the line of feasibility, choose the V1 scalable slice, and use Compound Engineering `ce-brainstorm` when research leaves multiple valid product shapes before specs become implementation plans.
7. **Phase 1d: paid user acquisition system** - `PAID_UA.md` ties one-channel paid acquisition, creative production, tracking baseline, blended reporting, RevenueCat economics, store destination, and founder spend gates together before paid-growth claims harden.
8. **Phase 1e: viral growth loop contract** - `VIRAL_GROWTH.md` ties the 11-star slice to product-led sharing/referrals, content format tests, monetization timing, analytics proof, abuse controls, and stop/scale rules before growth claims harden. `growth/LAUNCH_NARRATIVE.md` then plans the public launch as an event: feeling-first thesis and emotional angle, the tentpole-plus-weekly-cadence model, the launch-day run-of-show, and copy that passes the 2026 guardrails — refined again at Phase 3 (launch dossier) and Phase 6 (post-launch engine).
9. **Phase 1f: launch trace and build contracts** - `LAUNCH_TRACE.md` maps evidence to product, 11-star experience, paid UA, viral growth, brand/design, build, analytics, revenue, privacy, store, and verification decisions; `TECH_SPEC.md` is created when implementation needs data/API/state contracts.
10. **Phase 1g: security architecture** - threat model, `SECURITY.md`, security tool routing, OWASP MASVS/ASVS basis, platform hardening decisions, app-integrity/abuse posture, security agent role, and `security-review.html`.
11. **Phase 2: brand and design** - brand rules, `DESIGN.md` token/prose system, Higgsfield-backed visuals/motion/icons/mascots, Remotion-ready content asset rules when local rendering is selected, lowercase `design.md` screen spec, rendered HTML visual proofs, key generated or sourced assets, outreach drafts if partnerships matter.
12. **Phase 3: launch dossier and store ops** - ASO context, App Store/Play metadata, screenshots, paid/social ad library, Apple Search Ads, Apple Developer/App Store signing and upload readiness, App Store Connect/Play Console readiness, App Store listing packet, Apple pre-ASC requirements packet, privacy questionnaire, in-app event/custom product page plan, ASC CLI/skill routing for app creation, metadata, screenshots, TestFlight, review/status, products/subscriptions, and RevenueCat reconciliation, `APPLE_SIGNING.md`, `APPLE_APP_STORE_REQUIREMENTS.md`, `STORE_CONSOLE.md`, `store-console.html`, `APP_STORE_LISTING.md`, `CONTENT_ASSETS.md`, MobAI/Codex Desktop/XcodeBuildMCP/serve-sim screenshot matrix with SnapshotPreviews marked preview-only when used, ParthJadhav/app-store-screenshots or equivalent composition/export board, optional Remotion app previews/screenshot frames, review/rejection playbook, launch calendar, monetization metrics, open founder decisions.
13. **Phase 3b: revenue and monetization ops** - RevenueCat, app-store products, Stripe/Web Billing choice, web purchase links/funnels, paywall, entitlement identity, webhooks, pricing/legal gates, sandbox and production purchase validation.
14. **Phase 4: pre-launch funnel** - landing page, waitlist/referral loop, optional web monetization funnel, domain, routed support/privacy email aliases, Resend outbound/lifecycle email, Supabase/RPC or equivalent backend, PostHog/GA/Sentry, security headers, GEO/SEO, AI crawler access, robots/sitemap/llms/schema, live deploy verification.
15. **Phase 5: builder/agent handoff** - Rork/agent prompt sequence, `AGENTS.md`, `CLAUDE.md`, `APP_AGENTS.md`, `agents/`, `SECRETS.md`, `SECURITY.md`, `LAUNCH_TRACE.md`, `TECH_SPEC.md`, `ORCHESTRATION.md`, analytics catalog, V1/V2 scope locks, asset bundle, push/commit instructions, audit prompt.
16. **Phase 5b: engineering orchestration and production readiness** - Compound Engineering `ce-plan`/`ce-work`, orchestration preflight, safe parallel agents/worktrees, serialized shared resources, LaunchBench/validator checks, MobAI or native iOS device/simulator E2E through Codex Desktop/XcodeBuildMCP/serve-sim, SnapshotPreviews preview proof when useful, backend/frontend proof, secret injection proof, review, `ENGINEERING_PLAN.md`, `ORCHESTRATION.md`, and `PRODUCTION_READINESS.md`.
17. **Phase 5c: security release gate** - run local security validators/scans or founder-approved paid security routes, verify mobile/backend/provider controls, render `security-review.html`, resolve/accept risks, and attach proof to `PRODUCTION_READINESS.md`.
18. **Phase 6: post-launch UGC/Fastlane growth engine** - viral-loop fit refresh, UGC fit check, 90-day creator format-discovery plan, Fastlane workspace setup, social account connections, API key handling, Blitz angles/preferences, MobAI/native iOS/Higgsfield/Remotion media inputs, generated and rendered content QA, scheduling approvals, analytics snapshots, weekly iteration loop.
19. **Phase 6b: post-launch operations** - `POST_LAUNCH_OPS.md` weekly operating rhythm (crash triage with a named route and release gate, review responses with an SLA, release/hotfix cadence with staged rollout, retention review with cohort source and churn split, support operations), `LAUNCH_RETRO.md` at launch +7/30/90 days feeding failure cards and LaunchBench candidates, and the day-30 revisit of any lite-tier deferred lanes.
